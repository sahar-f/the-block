# The Block

Buyer-side vehicle auction prototype — OPENLANE coding challenge.

**Live demo:** https://the-block-red.vercel.app

For the full writeup (architecture, assumptions, decisions, testing, what's next), see [SUBMISSION.md](SUBMISSION.md).

## How to Run

The **live demo** is the intended evaluator experience — it runs against Supabase, so real-time bidding, cross-tab updates, and bid persistence all work out of the box. Two minutes on the live URL shows the app at full capability.

Local:

```bash
git clone https://github.com/sahar-f/the-block.git
cd the-block/app
npm install
npm run dev
```

A local run works with zero env vars — the app falls back to a committed JSON dataset (200 vehicles, AI condition summaries baked in). For the full feature set locally — real-time bid propagation, persistence across refreshes, atomic server-side bidding — provision Supabase and add `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` to `app/.env`. Schema and RPC live in [supabase/migrations/](supabase/migrations/); seed via [scripts/seed.mjs](scripts/seed.mjs).

## Architecture at a Glance

Four folders under [app/src/](app/src/) — `components/`, `hooks/`, `lib/`, `pages/` — each with a single clear responsibility. It's a shape that keeps cognitive load bounded while the app is small and composes cleanly as features are added: a new developer can orient themselves in five minutes by reading the tree and [CLAUDE.md](CLAUDE.md), and every feature has exactly one right place for each of its pieces. Premature feature-folders would have cost more than they'd have saved at this scale. Deeper architectural discussion is in [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) and [SUBMISSION.md](SUBMISSION.md).

## AI Workflow

Built with Claude Code using **Claude Opus 4.6** (architecture, planning, multi-round review, judgment calls) and **Sonnet** (mechanical implementation tasks dispatched to subagents).

### Skills

- **`brainstorming`** — forced early design alignment before code. Several feature cuts surfaced in design review rather than after implementation.
- **`writing-plans`** — every spec converted to an executable task list with exact file paths, commands, and code blocks.
- **`subagent-driven-development`** — fresh subagent per task, two-stage review (spec compliance then code quality). Preserved main-thread context for coordination.
- **`frontend-design`** — aesthetic direction: amber-on-dark palette, stats bar, ending-soon strip, card entrance animations, visual hierarchy.
- **`vercel-react-best-practices`** — React 19 patterns audit on TSX edits: component structure, hooks discipline, accessibility, performance.
- **`web-design-guidelines`** — UX and accessibility audit against Vercel's Web Interface Guidelines: focus rings, semantic landmarks, keyboard flow, `prefers-reduced-motion` respect, touch targets.
- **`test-driven-development`** — red/green/refactor on the bidding flow (highest-risk feature).
- **`using-git-worktrees`** — isolated workspaces for feature work; safe to experiment without contaminating main.
- **`systematic-debugging`** — structured approach for bug investigation (e.g., diagnosing the Supabase sync gap that hid the Condition Summary on realtime-backed pages).
- **`simplify`** — dead-code removal and unnecessary-abstraction pruning after each feature landed.
- **`requesting-code-review`** — security review, code-quality review, and simplification audit dispatched in parallel after every major feature.

### MCP servers

- **Context7** — live library docs for Vite 6, Tailwind 4, React Router v7, Supabase JS v2, Playwright.
- **Supabase** — project, schema, RLS policies, `place_bid` RPC, realtime publication config, migrations.
- **Vercel** — deploy, env vars, build log inspection; auto-deploy from `main`.
- **Playwright** — screenshot-driven E2E debugging.

### Pattern that shaped the build

**Brainstorm → spec → plan → subagent execution → multi-round review → simplify.** Each stage gated the next. Design reviews killed weak feature ideas before code was written. Task-level reviews caught real bugs (whitespace-only summary rendering an empty card, unnamed accessibility landmark, false-green null test, stale Supabase data hiding the AI card on the live demo). The simplification pass cut ~30% of lines from feature code without losing behavior.

### AI in the product

Beyond development tooling, AI is a product feature: every vehicle detail page shows a 2-sentence condition summary generated by **Claude Haiku 4.5** via [scripts/enrich.mjs](scripts/enrich.mjs). Pre-computed at build time, baked into the committed JSON, validated against length and content guards, idempotent across reruns. Paired with an outbound link to CarFax Canada's public market-value tool — the buyer gets an AI-synthesized condition read and an independent value-verification path without the prototype making any price claim itself.
